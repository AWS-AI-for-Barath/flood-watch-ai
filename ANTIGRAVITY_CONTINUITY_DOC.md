## 🏗️ 1. Project Architecture Overview

FloodWatch AI is a multimodal, serverless mapping and routing platform for flood disaster response. We have defined a 5-Phase architecture.

### Phase 1: Edge Ingestion & Community Input (✅ Built)
*   **Frontend:** A Progressive Web App (PWA) capturing 10-second videos alongside telemetry (GPS coordinates, heading, pitch/yaw).
*   **Backend:** Uploads go directly to an S3 bucket (`floodwatch-raw-images`) via pre-signed URLs generated by a Python Lambda (`lambda_presign`). This triggers an EventBridge rule.

### Phase 2: Multimodal AI & Computer Vision (🟡 Mocked / Next Step)
*   **The Plan:** Use Amazon Bedrock (Nova) for semantic context (e.g., are people trapped?) and Amazon SageMaker (YOLOv8/Segment Anything) to calculate exact water depth by analyzing the waterline against known objects (tires, doors).
*   **Current State:** There is a Node.js/Python stub `processFloodAI` that mocks this behavior. **Implementing Bedrock/SageMaker is a priority task in this new session.**
*   **Output:** Saves depth and severity JSON to `floodwatch-ai-predictions` S3 bucket.

### Phase 3: Live DTM Generation & GIS Integration (✅ Built)
*   **The Base:** PostGIS database (on RDS) holding bare-earth elevation models in `dem_table`.
*   **Transformation:** `transformFloodPolygon` Lambda triggers from S3. It queries the `dem_table` for ground elevation at the GPS point, adds the AI's water depth locally to find `water_surface_elevation`, and generates GeoJSON polygons inserted into the `flood_layer` table.

### Phase 4: Hydrodynamic Simulation & Predictive Routing (✅ Built but Needs Real Data)
*   **Routing:** `routingLambda` queries the **public OSRM API** (`router.project-osrm.org`) for road paths.
*   **Flood Propagation:** Python code uses PostGIS DEM data to predict where water will flow downhill.
*   **Risk Math:** Uses `Shapely` to calculate intersection between the OSRM road polyline and the PostGIS flood polygons, returning risk levels (`ok`, `rerouted`, `blocked`) and exposure length.

### Phase 5: Mass Alerting via Pinpoint (❌ Planned Future Expansion)
*   AWS Lambda will query a user database for phones within the flooded polygons.
*   Amazon Pinpoint + Polly will dispatch localized text-to-speech voice and SMS warnings.

## 🗄️ 2. Core Infrastructure & Database Blueprint

When standing up the new AWS account, you MUST deploy an **Amazon RDS PostgreSQL** instance with the **PostGIS extension** enabled. 

**CRITICAL:** The RDS instance MUST have **Public Access = Yes**. The `routingLambda` (Phase 4) cannot be placed inside a VPC, because if it is, it loses outward internet access and cannot hit the public OSRM API (unless you pay for an expensive NAT Gateway).

### Exact PostGIS Schema
Run this in DBeaver/pgAdmin on the new RDS instance:

```sql
CREATE EXTENSION IF NOT EXISTS postgis;

-- Phase 3: Observed Floods
CREATE TABLE IF NOT EXISTS flood_layer (
    id SERIAL PRIMARY KEY,
    geom geometry(Polygon, 4326),
    submergence_ratio DOUBLE PRECISION,
    severity VARCHAR(50),
    timestamp TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    water_surface_elevation DOUBLE PRECISION,
    source_image VARCHAR(255)
);
CREATE INDEX ON flood_layer USING GIST(geom);

-- Phase 3 & 4: DEM Terrain Baseline
CREATE TABLE IF NOT EXISTS dem_table (
    id SERIAL PRIMARY KEY,
    geom geometry(Point, 4326),
    elevation_m DOUBLE PRECISION
);
CREATE INDEX ON dem_table USING GIST(geom);

-- Phase 4: Predicted Flood Expansion
CREATE TABLE IF NOT EXISTS flood_prediction (
    id SERIAL PRIMARY KEY,
    geometry geometry(Polygon, 4326),
    submergence_ratio DOUBLE PRECISION,
    velocity DOUBLE PRECISION,
    timestamp TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP,
    source VARCHAR(50)
);
CREATE INDEX ON flood_prediction USING GIST(geometry);

-- Phase 4: Road Risk Cache
CREATE TABLE IF NOT EXISTS road_risk (
    id SERIAL PRIMARY KEY,
    road_segment_id VARCHAR(100) UNIQUE,
    geometry geometry(LineString, 4326),
    base_weight DOUBLE PRECISION,
    dynamic_weight DOUBLE PRECISION,
    max_submergence DOUBLE PRECISION,
    is_closed BOOLEAN,
    updated_at TIMESTAMP WITH TIME ZONE DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX ON road_risk USING GIST(geometry);
```

## ⚠️ 4. Historical Context & Critical Lessons Learned (DO NOT REPEAT THESE MISTAKES)

1. **Abandoned EC2 OSRM Deployment:** We spent days trying to run the OSRM Routing Engine on an EC2 instance. It crashed due to memory limits (`bad_alloc` on `t3.medium`). We upgraded to `t3.large` but hit persistent SSH key pairing issues and VPC subnet blockers. **Conclusion:** We migrated fully to querying the public internet API (`router.project-osrm.org`). Do not attempt to host OSRM on EC2 again unless the user explicitly demands it and provides a massive instance.
2. **Lambda VPC Internet Loss:** By putting `routingLambda` in a VPC to talk to RDS, it lost internet access to OSRM. **Conclusion:** Remove Lambda from the VPC. Make the RDS instance publicly accessible and secure it with a Security Group IP whitelist.
3. **Empty DEM Data = Failure:** The system relies entirely on the `dem_table` having real elevation points for hydrodynamic simulation. **Conclusion:** In the new account, you MUST download SRTM or Copernicus DEM data for your target city (e.g., Chennai) and write a script to insert it into `dem_table`.
